# 딸기 전기/펠릿 데이터셋 매핑

이 프로젝트는 스마트팜코리아에서 내려받은 `딸기 전기`, `딸기 펠릿` 폴더의 엑셀 데이터를 BerryNext 딸기 온실 의사결정 보조 시스템의 입력으로 사용한다.

## 폴더 구성

두 폴더는 같은 이름의 8개 `.xlsx` 파일을 가진다.

| 파일 | 용도 |
| --- | --- |
| `시설원예_생육정보_2026-06-23.xlsx` | 주차별 생육 상태 |
| `시설원예_시간기준_환경정보_2026-06-23.xlsx` | 시간 단위 온실환경 |
| `시설원예_일기준_환경정보_2026-06-23.xlsx` | 일 단위 온실환경 |
| `시설원예_주기준_환경정보_2026-06-23.xlsx` | 주 단위 온실환경 |
| `시설원예_월기준_환경정보_2026-06-23.xlsx` | 월 단위 온실환경 |
| `시설원예_시간대별_환경정보_2026-06-23.xlsx` | 야간/주간 등 시간대 평균 환경 |

## 통합 정리본 기준

로컬 자료는 다음 폴더를 기준으로 정리한다.

```text
C:\Users\연재성\Desktop\딸기_AI_의사결정_통합정리본
```

이 정리본은 딸기 전용 데이터만 남기는 것을 원칙으로 한다.

| 정리본 폴더 | 사용 여부 | 설명 |
| --- | --- | --- |
| `01_핵심_운영데이터` | 필수 | 전기/펠릿 실재배 데이터. MVP 1순위 입력이다. |
| `02_외부_분석데이터` | 보조 | 혼합 데이터는 딸기 행만 `*_딸기만.csv`로 분리했다. |
| `03_참고자료` | 보조 | 관수, EC/양액, 환기, 차광, 보온 기준과 Level 2 보조 알림 근거를 설명하는 원본 자료다. |
| `03_참고자료_브라우저용_HTML` | 열람용 | 원본 HTML이 깨질 때 보는 UTF-8 HTML 자료집이다. |
| `04_모델링_가이드` | 필수 | 인벤토리, 컬럼 요약, 작업별 자료 매핑, 삭제 로그를 둔다. |

정리본에서 제거한 항목은 다음과 같다.

- 딸기와 다른 작물이 섞인 대용량 지역 센서 로우데이터 CSV/ZIP
- `오이/딸기`처럼 다른 작물과 함께 제공되는 이미지 메타자료
- SmartFarmKorea 전체 품목 안내처럼 딸기 전용이 아닌 일반 페이지

행 단위로 분리 가능한 혼합 엑셀은 원본 `.xlsx` 대신 딸기 행만 추출한 `*_딸기만.csv`를 사용한다.

## 전기/펠릿 차이

전기와 펠릿은 공통 환경 컬럼이 많지만, 센서 구성이 완전히 같지 않다.

| 구분 | 주요 차이 |
| --- | --- |
| 딸기 전기 | `토양-지습`, `양액-지습`, `내부-내부일사량`, `이슬점온도`, `절대습도`가 있다. |
| 딸기 펠릿 | `토양-토양EC`, `토양-토양PH`, `양액-(양액)배액EC`, `양액-(양액)배액PH`가 있다. |
| 제어정보 | 전기는 보일러, 유동팬, CO2 발생장치 등 제어 컬럼이 많고 결측도 많다. 펠릿은 창 개도율과 차광커튼 중심으로 더 단순하다. |

따라서 모델 입력은 다음처럼 나눈다.

1. 공통 특징: 온도, 습도, CO2, 외부일사량, 외부온도, 강우감지, 풍속, 천창/측창 개도율
2. 전기 전용 특징: 지습, 내부일사량, 이슬점, 절대습도, 보일러/팬/CO2 제어 여부
3. 펠릿 전용 특징: 토양 EC, 토양 pH, 배액 EC, 배액 pH
4. 비교 특징: 난방방식 라벨 `electric` 또는 `pellet`

## 오늘 온실 관리 의사결정 구조

새 추천 파이프라인은 하루 단위로 정규화한 현재 상태를 `GreenhouseSnapshot` 계약으로 맞춘 뒤 Level 1 작업과 Level 2 보조 알림을 분리한다. 구형 `DailyFarmWorkDecisionEngine`용 JSON은 호환 예제로 남아 있지만, 최종 방향은 아래 구조다.

| 구분 | 항목 | 판단 근거 |
| --- | --- | --- |
| Level 1 | 관수 | 고온, 고일사, 낮은 지습, 높은 VPD, 예측 수분 감소, 최근 관수 안전 간격 |
| Level 1 | EC/양액 조정 검토 | 급액/배액 EC, 토양/근권 EC, pH, 배액률, 최근 양액 작업 안전 간격 |
| Level 1 | 환기 | 고습, 낮은 VPD, 외기 조건, 강우감지, 개도율, 예측 습도/VPD |
| Level 1 | 차광 | 고온, 고일사, 높은 VPD, 예측 온도/일사 상승 |
| Level 1 | 보온/난방 검토 | 내부 저온, 외기 저온, 야간 저온 예측, 난방방식 |
| Level 2 | 병해 위험 예찰 알림 | 고습, 낮은 VPD, 강우감지, 환기 부족, 병반 proxy. 실제 병해 예측이나 방제 처방이 아니다. |
| Level 2 | 수확 가능성 알림 | 생육 주차, 화방착과수, 가격 추세, 착색 proxy. 검증된 수확 시기 예측이 아니다. |
| Level 2 | 적엽 검토 알림 | 엽수, 엽장/엽폭, 고습, 환기 부족, 최근 적엽 안전 간격. 검증된 적엽 처방이 아니다. |

현재 폴더에는 직접 이미지 파일이 없으므로 생육정보의 엽수/엽장/엽폭/화방착과수와 환경 기반 병해 위험 proxy를 Level 2 보조 알림 신호로만 사용한다. 나중에 이미지 파일이나 병반 검출 결과가 추가되더라도 검증 전에는 보조 alert로만 표시한다.

## 전처리 단위

초기 MVP는 일 단위 데이터셋을 기준으로 만든다.

1. `일기준_환경정보`에서 날짜별 온실환경을 가져온다.
2. `생육정보`는 가장 가까운 조사일을 해당 날짜에 forward-fill한다.
3. `시간기준_제어정보`는 날짜별 평균/최대 개도율과 장비 작동 여부로 집계한다.
4. 작업이력은 지도학습 label이 아니라 안전 간격, cooldown, 설명을 위한 보조 정보로만 계산한다.
5. 결과는 우선 `examples/sample_daily_context.json` 같은 호환 JSON으로 저장하고, 새 파이프라인에서는 `decision_contract.GreenhouseSnapshot` 형태로 정규화한다.
6. `python3 -m examples.build_daily_context path/to/daily_context.json`는 legacy 호환 예제로 유지한다. 최종 추천은 `ActionRecommendationEngine`과 `ScenarioSimulator` 중심으로 정렬한다.

## 모델 확장 방향

처음에는 문헌/매뉴얼 규칙과 baseline 예측으로 오늘의 온실 관리 필요도를 설명 가능하게 추천한다. 데이터가 더 쌓이면 직접 작업 분류가 아니라 단기 환경 변화량 예측기를 단계적으로 개선한다.

| 단계 | 방법 |
| --- | --- |
| v0 | no-change baseline |
| v1 | rolling delta baseline |
| v2 | linear regression |
| v3 | GAM 기반 단기 환경 변화량 예측 |
| v4 | 충분한 데이터와 검증이 있을 때 LSTM 또는 Transformer 검토 |
