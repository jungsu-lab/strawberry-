# 딸기 AI 자료 정리본 운영 기준

이 문서는 로컬에 정리한 `딸기_AI_의사결정_통합정리본`을 BerryNext 딸기 온실 의사결정 보조 시스템 개발에 어떻게 연결할지 정리한다.

원본 데이터와 논문 파일은 용량과 라이선스, 로그인/신청 조건이 있으므로 GitHub에 커밋하지 않는다. GitHub에는 데이터 구조, 사용 우선순위, 전처리 기준만 기록한다.

## 딸기 전용 정리 원칙

정리본에는 현재 상태 구성, 단기 환경 예측, 문헌/매뉴얼 규칙 설명에 필요한 딸기 자료만 남긴다. 작업이력은 지도학습 label이 아니라 안전 간격, cooldown, 설명을 위한 보조 정보로만 사용한다.

1. 파일명 또는 데이터셋 자체가 딸기 전용이면 유지한다.
2. `품목`, `품목명`, `품종명` 컬럼으로 딸기 행을 분리할 수 있으면 `*_딸기만.csv`로 변환한다.
3. 여러 작물이 섞인 대용량 원본 CSV/ZIP은 정리본에서 제거한다.
4. `오이/딸기`처럼 다른 작물과 함께 제공되는 이미지 메타자료는 제거한다.
5. SmartFarmKorea 전체 품목 안내처럼 딸기 전용이 아닌 일반 페이지는 제거한다.
6. 딸기 논문 본문 안의 비교 문헌에 다른 작물명이 등장하는 것은 데이터 오염으로 보지 않는다.

## 모델 입력 우선순위

### 1순위: 전기/펠릿 실재배 데이터

`01_핵심_운영데이터`의 16개 엑셀 파일을 우선 사용한다.

- 생육정보: 생육단계, 엽수, 엽장, 엽폭, 화방/착과 신호
- 시간/일/주/월 환경정보: 온도, 습도, CO2, 일사, 강우, 풍속, VPD 파생값
- 시간 제어정보: 천창, 측창, 차광, 팬, 보일러, CO2 발생장치, 양액 관련 제어 이력
- 도매시장 가격정보: 수확 가능성 보조 알림과 출하 경제성 참고

### 2순위: 딸기 전용 외부 분석 데이터

`02_외부_분석데이터`를 기준값 보정과 사전 분석에 사용한다.

- `작기_생육환경출하`: 생육, 출하량, 환경 기준
- `복합_적산_환경지표`: 생육단계와 성숙 속도 피처 후보
- `지역_통계_센서`: 딸기만 추출한 지역/품목별 통계

### 3순위: 참고자료

규칙 기반 MVP의 기준값과 설명 근거로 쓴다.

- 관수/EC/양액: 일사, 배지수분, EC/pH, 배액률 판단 기준
- 환기/차광/보온: 습도, VPD, 고온, 저온, 일사, 외기 조건 기준
- 병해 예찰 보조 알림: 탄저병, 점박이응애, 고습/결로 위험 기준. 실제 병해 예측이나 방제 처방으로 쓰지 않는다.
- 수확 가능성 보조 알림: 설향 수확 성숙도, 품질, 가격 API 문서. 검증된 수확 시기 예측으로 쓰지 않는다.
- 적엽 검토 보조 알림: 설향 적엽 수준과 수량/품질 영향. 검증된 적엽 처방으로 쓰지 않는다.

| 구분 | 항목 | 1차 데이터 | 보조 데이터 | 추천 출력 |
| --- | --- | --- | --- | --- |
| Level 1 | 관수 | 시간 환경, 시간 제어, 근권 수분 | 관수 논문, 설향 수분 자료 | 관수 검토 필요도, 수분 부족/과습 주의사항 |
| Level 1 | EC/양액 조정 검토 | 근권/배액/급액 EC, pH, 배액률 | 설향 EC 자료, 양액 관리 자료 | EC/양액 점검 필요도, 조정 전 확인 항목 |
| Level 1 | 환기 | 습도, VPD, 외기온, 강우, 개도율 | 고습/결로 자료 | 환기/제습 검토 필요도, 냉해/건조 위험 |
| Level 1 | 차광 | 내부 온도, 일사, VPD | 고온/일사 관리 자료 | 차광/열부하 완화 검토 필요도, 과도 차광 주의 |
| Level 1 | 보온/난방 검토 | 내부 온도, 외기온, 난방방식 | 저온 관리 자료 | 보온/난방 점검 필요도, 에너지/습도 주의 |
| Level 2 | 병해 위험 예찰 알림 | 습도, VPD, 강우, 환기 이력 | RDA/농사로 병해충 자료, PSIS | 예찰 우선순위. 실제 병해 예측이나 방제 처방이 아니다. |
| Level 2 | 수확 가능성 알림 | 생육정보, 도매가격 | 출하량, 수확 성숙도 연구 | 수확 가능성 모니터링. 검증된 수확 시기 예측이 아니다. |
| Level 2 | 적엽 검토 알림 | 생육정보, 습도/VPD, 최근 적엽 안전 간격 | 적엽 연구자료 | 보수적 적엽 검토. 검증된 적엽 처방이 아니다. |

## GitHub에 올리지 않는 것

- 로컬 엑셀, CSV, HWP, PDF, HTML 원본 파일
- AI-Hub 원천 이미지와 JSON 라벨
- SmartFarmKorea 로그인 후 받은 ZIP/CSV 원본
- KAMIS 또는 공공데이터 API 키

이 저장소에는 입력 스키마, 매핑 문서, 의사결정 엔진, 예제와 테스트만 유지한다.

## 다음 개발 순서

1. `01_핵심_운영데이터`를 일 단위 JSON으로 집계한다.
2. `examples/sample_daily_context.json`과 같은 스키마로 저장한다.
3. 새 추천 파이프라인에서는 `GreenhouseSnapshot`, `PredictionResult`, `EvidenceRule`, `RecommendationResult` 계약으로 연결한다.
4. 실제 작업 로그를 추가하더라도 추천 결과의 안전 간격, cooldown, 설명에만 사용한다.
5. 작업 분류 모델이 아니라 단기 환경 변화량 예측기를 v0 no-change, v1 rolling delta, v2 linear regression, v3 GAM, v4 LSTM/Transformer 순서로 검토한다.

## 샘플 JSON 계약

GitHub에는 원본 데이터 대신 하루치 샘플만 둔다.

```text
examples/sample_daily_context.json
```

필수 상위 키는 다음과 같다.

| 키 | 설명 |
| --- | --- |
| `growth_stage` | `transplanting`, `vegetative`, `flowering`, `fruiting`, `harvest`, `resting` 중 하나 |
| `snapshot` | 하루 단위로 집계한 온실환경, 양액/근권, 이미지/예찰 신호 |
| `history` | 최근 작업 후 경과일. 지도학습 label이 아니라 안전 간격, cooldown, 설명용 정보 |

전기/펠릿 엑셀 전처리기는 우선 이 JSON을 만들 수 있다. 최종 파이프라인에서는 같은 정보를 `decision_contract.GreenhouseSnapshot` 형태로 정규화해 Level 1 관리 작업과 Level 2 보조 알림을 분리한다.
