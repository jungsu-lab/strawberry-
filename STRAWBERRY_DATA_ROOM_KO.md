# 딸기 AI 자료 정리본 운영 기준

이 문서는 로컬에 정리한 `딸기_AI_의사결정_통합정리본`을 BerryNext 농작업 의사결정 AI 개발에 어떻게 연결할지 정리한다.

원본 데이터와 논문 파일은 용량과 라이선스, 로그인/신청 조건이 있으므로 GitHub에 커밋하지 않는다. GitHub에는 데이터 구조, 사용 우선순위, 전처리 기준만 기록한다.

C:\Users\연재성\Desktop\딸기_AI_의사결정_통합정리본
```

WSL 기준:

```

## 핵심 폴더

| 폴더 | 용도 |
| --- | --- |
| `01_핵심_운영데이터` | 딸기 전기/펠릿 재배 실데이터. MVP의 1순위 입력이다. |
| `02_외부_분석데이터` | 작기, 생육, 환경, 출하, 적산온도, 지역 통계 보조 데이터다. 혼합 데이터는 딸기 행만 `*_딸기만.csv`로 분리했다. |

| `03_참고자료_읽기용_TXT` | 본문 검색과 복사를 위한 TXT 자료다. |
| `04_모델링_가이드` | 데이터 인벤토리, 컬럼 요약, 작업별 자료 매핑, 삭제 로그다. |
| `99_원본_보관` | 다운로드 인덱스만 남긴다. 여러 작물이 섞인 대용량 원본은 정리본에서 제거했다. |

## 딸기 전용 정리 원칙

정리본에는 모델 학습과 설명에 필요한 딸기 자료만 남긴다.

1. 파일명 또는 데이터셋 자체가 딸기 전용이면 유지한다.
2. `품목`, `품목명`, `품종명` 컬럼으로 딸기 행을 분리할 수 있으면 `*_딸기만.csv`로 변환한다.
3. 여러 작물이 섞인 대용량 원본 CSV/ZIP은 정리본에서 제거한다.
4. `오이/딸기`처럼 다른 작물과 함께 제공되는 이미지 메타자료는 제거한다.
5. SmartFarmKorea 전체 품목 안내처럼 딸기 전용이 아닌 일반 페이지는 제거한다.
6. 딸기 논문 본문 안의 비교 문헌에 다른 작물명이 등장하는 것은 데이터 오염으로 보지 않는다.

## 모델 입력 우선순위

### 1순위: 전기/펠릿 실재배 데이터

`01_핵심_운영데이터`의 16개 엑셀 파일을 우선 사용한다.

- 생육정보: 생육단계, 엽수, 엽장, 엽폭, 화방/착과 신호
- 시간/일/주/월 환경정보: 온도, 습도, CO2, 일사, 강우, 풍속, VPD 파생값
- 시간 제어정보: 천창, 측창, 차광, 팬, 보일러, CO2 발생장치, 양액 관련 제어 이력
- 도매시장 가격정보: 수확/출하 경제성 판단

### 2순위: 딸기 전용 외부 분석 데이터

`02_외부_분석데이터`를 기준값 보정과 사전 분석에 사용한다.

- `작기_생육환경출하`: 생육, 출하량, 환경 기준
- `복합_적산_환경지표`: 생육단계와 성숙 속도 피처 후보
- `지역_통계_센서`: 딸기만 추출한 지역/품목별 통계

### 3순위: 참고자료

규칙 기반 MVP의 기준값과 설명 근거로 쓴다.

- 관수/양액: 일사, 배지수분, EC/pH, 배액률 판단 기준
- 방제/예찰: 탄저병, 점박이응애, 고습/결로 위험 기준
- 수확/가격: 설향 수확 성숙도, 품질, 가격 API 문서
- 적엽: 설향 적엽 수준과 수량/품질 영향

## 작업별 연결

| 작업 | 1차 데이터 | 보조 데이터 | 추천 출력 |
| --- | --- | --- | --- |
| 관수/양액 | 시간 환경, 시간 제어, 근권/EC/pH | 관수 논문, 설향 EC 자료 | 관수 필요도, 오늘 관수 주의사항, EC/pH 점검 |
| 방제/예찰 | 습도, VPD, 강우, 환기 이력 | RDA/농사로 병해충 자료, PSIS | 예찰 우선순위, 방제 검토, 안전사용 확인 |
| 수확 | 생육정보, 도매가격 | 출하량, 수확 성숙도 연구 | 수확 우선구역, 수확 가능성, 가격 판단 |
| 적엽 | 생육정보, 습도/VPD, 최근 적엽 이력 | 적엽 연구자료 | 적엽 필요도, 과도 적엽 방지 주의 |

## GitHub에 올리지 않는 것

- 로컬 엑셀, CSV, HWP, PDF, HTML 원본 파일
- AI-Hub 원천 이미지와 JSON 라벨
- SmartFarmKorea 로그인 후 받은 ZIP/CSV 원본
- KAMIS 또는 공공데이터 API 키

이 저장소에는 입력 스키마, 매핑 문서, 의사결정 엔진, 예제와 테스트만 유지한다.

## 다음 개발 순서

1. `01_핵심_운영데이터`를 일 단위 `FarmWorkContext`로 변환하는 전처리기를 만든다.
2. 전기/펠릿 데이터를 같은 스키마로 정규화한다.
3. `DailyFarmWorkDecisionEngine.plan_today()`로 날짜별 추천 로그를 생성한다.
4. 실제 작업 로그를 추가해 추천 결과와 실제 농작업/수확 결과를 비교한다.
5. 작업별 점수 함수를 학습 모델로 교체한다.
